# v93 — Performance Pass

Цель прохода: убрать самые опасные тормоза перед публичным запуском без тяжёлой архитектурной переделки.

## Что изменено

### 1. Списки постов больше не тянут все комментарии

Для ленты, профиля, чужого профиля и постов сообщества теперь используется preview-include:

- последние комментарии ограничены небольшим лимитом;
- точное количество комментариев берётся через `_count`;
- точное количество сохранений берётся через `_count`;
- `saves` грузятся только для текущего пользователя, чтобы определить `is_saved`;
- `votes` грузятся минимальным select-набором: `userId` и `value`.

Это сохраняет текущий UX карточек, но не превращает каждую страницу в тяжёлый запрос при росте данных.

### 2. Добавлена простая cursor-pagination

Поддержаны параметры:

- `limit`;
- `cursor` как ISO-дата `createdAt`.

Затронутые endpoints:

- `GET /api/feed`;
- `GET /api/profile/posts`;
- `GET /api/users/:id/posts`;
- `GET /api/profile/media`;
- `GET /api/users/:id/media`;
- `GET /api/communities/:slug/posts`;
- `GET /api/communities/:slug/media`.

Ответы теперь могут содержать блок:

```json
{
  "page": {
    "limit": 30,
    "has_more": true,
    "next_cursor": "2026-04-26T12:00:00.000Z"
  }
}
```

Старые клиенты могут продолжать читать `posts`, `items` или `media`; блок `page` добавлен совместимо.

### 3. Медиа-списки стали легче

Медиа endpoints теперь используют `select` вместо широкого `include`, чтобы не тянуть лишние связи и поля.

### 4. Добавлены DB indexes

Миграция:

- `prisma/migrations/20260426_performance_pass/migration.sql`.

Добавлены индексы для горячих сценариев:

- feed/profile/community post lists;
- media lists;
- comment preview loading;
- community membership checks.

### 5. Добавлен performance check

Команда:

```bash
npm run performance:check
```

Проверяет наличие:

- performance helper;
- migration;
- индексов;
- preview-comments подхода;
- pagination-полей;
- документации.

## Что намеренно не делал в v93

- не добавлял виртуализацию UI без реальных замеров;
- не добавлял Redis/cache-layer;
- не переписывал ленту на сложный ranking engine;
- не менял публичный контракт карточек постов;
- не делал сложные aggregate-запросы для голосов, чтобы не раздувать проход.

## Что проверить на реальной базе

1. `/api/feed?limit=30` возвращает первый экран быстро.
2. `page.next_cursor` можно передать в следующий запрос.
3. Количество комментариев/сохранений на карточке остаётся корректным.
4. Превью комментариев показывает только последние комментарии.
5. Community media и profile media не тянут лишние связи.
6. `EXPLAIN ANALYZE` на Post/Comment запросах использует новые индексы.

## Известное ограничение

Счётчики плюсов/минусов пока считаются по загруженным votes, но payload стал легче за счёт `select`. Если на публичной нагрузке голосов будет очень много, следующий шаг — денормализованные счётчики `plusCount/minusCount` в `Post` и `Comment`.
